# 反模式：用浮点增量累积媒体时间轴

## 一句话结论

长时音视频时间轴应从绝对 host time 映射到固定有理 timescale，不能逐回调累加 `Double` 秒数；否则舍入误差、样本边界和停止裁剪会在不同路径上产生不一致。

## 场景

视频、系统音频和麦克风共享一个 `t0`/`t1`，音频输出为 48 kHz，视频以 `CMTime` 写入。

## 失败表现

- 刚好跨过 `t0` 的音频块多裁或少裁一帧。
- 同一个 host timestamp 经不同代码路径得到不同输出 PTS。
- 长时间运行后，累计帧数与目标时长偏离。
- `t1` 边界处出现负时间、重叠或一帧空洞。

## 根因

逐段执行 `elapsed += frames / sampleRate` 会反复舍入。若视频、系统音频和麦克风各自维护累计量，就没有一个可证明的一致时间原点。

## 正确做法

- 只保存一个精确 `recordingStartHostTime`。
- 每次用 `hostTime - t0` 的绝对差值映射到输出 timescale。
- 音频 timescale 直接使用输出采样率，例如 48,000。
- `CMTimeConvertScale` 使用明确的舍入模式。
- 跨 `t0` 的音频起始裁剪向远离零方向取整，确保第一个接受样本不早于 `t0`。
- 返回源帧偏移、接受帧数和输出起始帧，而不是模糊的秒数。

## 验证清单

- 覆盖 `t0-1 sample`、恰好 `t0`、`t0+1 sample`。
- 跨界 buffer 验证裁剪后的首帧 host time 不早于 `t0`。
- 60 分钟和 ±ppm 合成用例检查最终帧数不超过一帧误差。
- 非单调、无效、pre-start timestamp 明确拒绝或返回 nil。

## 适用范围

适用于录音、录屏、转码、传感器融合等以离散样本构造统一时间轴的系统。
