# 反模式：忽略大型证据却不提交可审计清单

## 一句话结论

MP4、PNG、日志等证据不适合进 Git 时，提交文档必须固定代码身份、运行 ID 和每个证据文件的 SHA-256，并诚实说明证据只保存在本机。

## 场景

实机验证生成多个二进制 bundle，`Artifacts/` 被 `.gitignore` 排除。报告只提交“六次都通过”和相对目录路径。

## 失败表现

当前机器上能验证，换一台机器或工作树清理后：

- 不知道当时验证的是哪六个目录。
- MP4/PNG 可能被替换但文档仍显示 PASS。
- `report.json` 或 `probe.log` 被修改后无法证明它们是评审时看到的版本。
- 最终 rebuild 的二进制与 capture-time 二进制身份不明。

## 根因

把“文件可访问性”和“文件内容身份”混为一谈。相对路径只能定位文件，不能固定内容；报告内只哈希媒体也无法固定报告自身和日志。

## 正确做法

提交一个紧凑 manifest，每次运行至少记录：

| 字段 | 作用 |
| --- | --- |
| 代码 commit | 固定实现 |
| capture-time executable SHA-256 | 固定运行二进制 |
| run ID | 固定一次运行 |
| MP4 SHA-256 | 固定视频 |
| PNG SHA-256 | 固定首帧 |
| report.json SHA-256 | 固定结构化结论 |
| probe.log SHA-256 | 固定运行日志 |

同时声明：

- 二进制证据是否进入版本控制。
- 本机保留目录。
- 是否存在不可变外部归档。
- fresh clone 是否需要另行复制证据。
- 测试命令 transcript 是否被保留和哈希。

## 验证清单

- 清单哈希由独立命令重新计算，不照抄报告。
- capture-time 和 final-package 二进制分别哈希并比较。
- 清单覆盖 bundle 内所有声称被评审的文件。
- 清理工作树前先复制并 `diff -qr` 验证证据目录。
- 不把“有哈希”夸大成“有外部不可变存储”。
