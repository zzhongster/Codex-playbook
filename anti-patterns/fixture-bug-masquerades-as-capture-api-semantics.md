# 反模式：验证夹具 bug 冒充底层 API 语义

## 一句话结论

当实机像素与坐标模型冲突时，先用独立测量证明“被拍对象确实在预期位置”；否则会把夹具偏移误诊为捕获 API 的坐标语义，继而修坏正确的生产代码。

## 场景

截图、录屏、OCR、视觉回归等系统常用自绘网格、色块、角标或测试窗口作为 ground truth。结果异常时，生产代码和验证夹具都可能有错。

## 失败表现

一次区域录屏验证中，本地 `sourceRect` 在外屏看到 790 像素“错误内容”和 333 像素纵向偏移。团队据此把 `sourceRect` 改成全局坐标；新结果变成 810 像素内容加约 790 像素黑边，问题更怪。

两组异常都有精确算术关系，但真正错误不是 ScreenCaptureKit，而是验证窗口把外屏原点加了两次。

## 根因

验证链共享了同一个未经证明的假设：

~~~text
预期选区坐标
  -> 绘制夹具
  -> 放置夹具窗口
  -> 捕获
  -> 分析像素
~~~

只检查最后的像素，无法区分“捕获错了”和“夹具一开始就放错了”。

## 正确做法

把链路拆成可独立观测的断点：

1. 读取并记录 AppKit 屏幕 frame。
2. 读取真实窗口 frame，证明夹具位置。
3. 读取捕获 filter 的 `contentRect` 和像素缩放。
4. 分别记录逻辑点源矩形、物理像素矩形、编码尺寸。
5. 一次只改变一个假设。
6. 新证据反驳模型时立即停止，不继续批量取证。

只有夹具位置已被独立证明，像素结果才能用于推断捕获 API 语义。

## 验证清单

- 夹具全局 frame 有运行时记录。
- 捕获画布 frame/scale 有独立记录。
- 错误偏移能否由夹具原点解释。
- 生产模型修改前有第二个对照实验。
- 错误实验保留，不能被成功重试覆盖。

## 适用范围

适用于所有“系统输出由自造 fixture 验证”的场景，不限于录屏。
