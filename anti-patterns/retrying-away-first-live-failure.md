# 反模式：重试掉第一份实机失败证据

## 一句话结论

实机验收必须使用全新证据根、逐次检查、第一次失败立即停止并保留；“再跑一次看看”会把确定性缺陷伪装成偶发波动。

## 场景

需要在两块显示器上顺序执行六组区域录屏，并汇总为一个 PASS_CURRENT_TOPOLOGY 结论。

## 失败表现

- 失败目录被删除或覆盖。
- 同一选区运行多次，只把成功那次写进表格。
- 前三组来自旧二进制，后三组来自修复后构建。
- 最终根目录里有七个或更多报告，却只选择六个“好看的”。

## 根因

批量验收被当成调试循环。调试允许重复试验，正式证据集则要求完整、可解释、同一版本、无选择性排除。

## 正确做法

1. 预检拓扑、权限、签名、运行进程和二进制哈希。
2. 创建此前不存在的唯一根目录。
3. 每条命令运行一次。
4. 每次结束立即验证 report、媒体、PNG 和哈希。
5. 任意门槛失败：保存现场、停止后续运行、不提交 PASS。
6. 修复后另建新根，从第 1 组完整重跑。
7. 最终要求目录数和报告数精确等于计划次数。

## 验证清单

- 根目录创建时间晚于被验证 commit。
- 运行时间严格递增。
- 没有 `-failed`、重试或第七个目录。
- 所有行来自同一代码 commit 和同一二进制哈希。
- 历史失败写入诊断，但不计入最终 verdict。
